Protocol Specification

GRAFT

Generic Remote Actuator Firmware Transport

A transport-agnostic, capability-first binary protocol for extensible host-device hardware control. Runs on any microcontroller. Travels over serial, BLE, MQTT, WebSocket, or CLASP. Speaks to any host SDK. Owned by no cloud.

v0.1 Draft Arduino / C++ Binary Protocol COBS Framing Transport Agnostic MQTT Native Self-Describing Open Protocol
01

What is GRAFT

GRAFT is an open binary protocol and firmware library for extending hardware control from a host to a connected microcontroller device. It replaces Firmata as the wire protocol, Johnny-Five as the abstraction layer, and Blynk as the IoT connectivity glue — without coupling the device to any cloud platform.

A device running GRAFT firmware self-describes its capabilities when connected. The host discovers everything — pins, buses, modules, datastreams — from the device itself. No cloud template configuration, no pre-registration, no hardcoded assumptions.

The Problem Space

ProtocolReleasedWhat it got wrong
Firmata2006Pseudo-binary, no real capability negotiation, serial-only, no module extensibility
Tentacle / Octoblu~2014Right idea (MQTT transport), wrong execution (Firmata underneath, coupled to dead cloud platform)
Johnny-Five2012Good host SDK DX, inherits all of Firmata's limitations, CommonJS/Node-only
Blynk 2.02021Solved the right problems, proprietary cloud required, string-typed virtual pins, template-first workflow
CircuitPython2017Runtime not protocol, device-as-computer model, no clean host→device control story

Design Goals

G1
Transport AgnosticSame binary framing over serial, BLE GATT, WebSerial, WebUSB, CLASP relay, raw TCP, or USB HID. Transport is a replaceable plug.
G2
Capability FirstDevice speaks first. Host never assumes. HELLO_RESP is the source of truth for everything the device has.
G3
Lightweight BinaryNo JSON, no text parsing on the MCU. Fixed-overhead packets. Runs on ATmega328 at 115200 baud without dropping bytes.
G4
Module SystemLow-level (raw pin r/w) and high-level (servo, OLED, NeoPixel) coexist as opt-in device-side modules.
G5
Bidirectional EventsDevice pushes data without polling. Interrupt pins, sensor streams, encoder ticks all fire unsolicited events.
G6
SEQ + ACKReliable messaging without requiring a reliable transport. Works on BLE where packet drops happen.
G7
No Cloud DependencyFirmware never phones home. No hardcoded endpoints. Auth is the broker's problem. Protocol is self-hostable by design.
G8
Named Typed DatastreamsBlynk-style virtual channel abstraction, but device-declared, binary-typed, and schema-described in HELLO_RESP.

Packet Structure

Every GRAFT packet on the wire follows a fixed header + variable payload structure. COBS encoding wraps the entire packet for serial/BLE transport, enabling 0x00 as a reliable packet delimiter.

Frame Layout

MAGIC 0x47 0x46 2 bytes
VER 0x01 1 byte
TYPE cmd/evt 1 byte
SEQ 0–255 1 byte
LEN uint16_le 2 bytes
PAYLOAD type-specific N bytes
CRC8 Dallas/Maxim 1 byte

Field Definitions

FieldSizeDescription
MAGIC2 bytes0x47 0x46 ("GF"). Catches misaligned reads on serial streams.
VER1 byteProtocol version. Currently 0x01. Device rejects mismatched versions with NAK 0x01.
TYPE1 byteCommand (host→device) or event (device→host) type identifier.
SEQ1 byte0–255 rolling sequence. Host increments per command. Device echoes in ACK/NAK. Wraps on overflow.
LEN2 bytesPayload length in bytes, little-endian uint16. Max 65535; device caps at max_payload from HELLO_RESP.
PAYLOADN bytesType-specific data. Zero-length for PING, RESET, etc.
CRC81 byteDallas/Maxim CRC8 computed over bytes [VER..end of PAYLOAD]. Mismatch returns NAK 0x02.

COBS Framing

For serial and BLE transports, each complete packet is wrapped in COBS (Consistent Overhead Byte Stuffing) encoding. This eliminates 0x00 from the payload data, making it usable as a packet delimiter with one byte overhead per 254 bytes of data. TCP and WebSocket transports carry packets as binary messages — COBS is optional but recommended for consistency.

06

HELLO_RESP

On connect, the host sends HELLO. The device replies with a structured binary payload declaring everything it has. The host builds a live capability model from this and never sends a command requiring a feature that isn't listed. HELLO_RESP is the single source of truth — not a cloud template, not a config file.

Payload Schema

// HELLO_RESP binary layout (little-endian)
firmware_name      : 16 bytes  // null-padded UTF-8
firmware_version   : 3 bytes   // major, minor, patch
mcu_id             : 8 bytes   // unique device ID, 0-padded
ota_capable        : uint8    // 0x01 if OTA supported
pin_count          : uint8
pins[]             : uint8[pin_count] // per-pin capability bitmask
  bit 0 = DIGITAL_IN
  bit 1 = DIGITAL_OUT
  bit 2 = PWM_OUT
  bit 3 = ANALOG_IN
  bit 4 = I2C_SDA
  bit 5 = I2C_SCL
  bit 6 = SPI
  bit 7 = INTERRUPT
i2c_buses          : uint8
spi_buses          : uint8
uart_count         : uint8
max_payload        : uint16   // device RAM budget per packet
module_count       : uint8
modules[]          : per-module descriptor
  module_id        : uint8
  name             : 8 bytes  // null-padded UTF-8
  version          : 2 bytes  // major, minor
  pin_count        : uint8
  pins[]           : uint8[]  // which physical pins this module claims
datastream_count   : uint8
datastreams[]      : per-datastream descriptor (see §08)

HELLO_RESP is published to the broker as a retained message on graft/{deviceId}/hello. Any host connecting after the device gets the full capability advertisement immediately, with no round-trip required.

07

Datastreams

Datastreams are named, typed, device-declared channels for higher-level data exchange. They are the GRAFT equivalent of Blynk's Virtual Pins, but with typed binary payloads and declared in the device's own HELLO_RESP — not configured in a cloud console.

A servo's angle is a writable FLOAT32 datastream named "angle". A temperature sensor is a read-only FLOAT32 named "temperature". An IMU exposes six read-only FLOAT32 datastreams. Everything is declared by the device at boot.

Datastream Descriptor (in HELLO_RESP)

datastream descriptor:
  name      : 16 bytes  // null-padded UTF-8, e.g. "temperature\0\0\0\0\0"
  type      : uint8    // see type codes below
  unit      : 8 bytes  // null-padded UTF-8, e.g. "celsius\0"
  writable  : uint8    // 0x01 = host can write, 0x00 = read-only
  pin_ref   : uint8    // physical pin mapping, 0xFF if none
  retain    : uint8    // 0x01 = broker retains last value (for state sync)

// Type codes
0x01 = BOOL      // 1 byte, 0 or 1
0x02 = INT8      // 1 byte signed
0x03 = UINT8     // 1 byte unsigned
0x04 = INT16     // 2 bytes, little-endian
0x05 = UINT16    // 2 bytes, little-endian
0x06 = INT32     // 4 bytes, little-endian
0x07 = FLOAT32   // 4 bytes, IEEE 754
0x08 = STRING    // uint16 length prefix + UTF-8 bytes
0x09 = BYTES     // uint16 length prefix + raw bytes

Firmware Declaration

// Declare datastreams in setup() before device.begin()
device.addDatastream("temperature", GRAFT_FLOAT32, "celsius", false);
device.addDatastream("setpoint",    GRAFT_FLOAT32, "celsius", true);
device.addDatastream("relay",       GRAFT_BOOL,    "",         true);

// Respond to host writes on writable datastreams
GRAFT_ON_DATASTREAM("setpoint") {
  float val = payload.readFloat32();
  pid.setTarget(val);
  graft.ack();
}

// Push datastream values from the device
void loop() {
  device.poll();
  if (tempTimer.elapsed()) {
    device.writeDatastream("temperature", sensor.readTemp());
  }
}

State Sync

All writable datastreams with retain = 0x01 have their last value retained at the broker. On device reconnect, the firmware automatically requests retained state for all retained writable datastreams before entering the main loop. Dashboard state survives device restarts without any explicit sync calls.

08

Subscribe & Stream

PIN_SUBSCRIBE Payload

// PIN_SUBSCRIBE payload
pin_num      : uint8
mode         : uint8    // 0x01=CHANGE, 0x02=RISING, 0x03=FALLING, 0x04=ANALOG_POLL
interval_ms  : uint16  // poll interval for analog (ignored for interrupt modes)
threshold    : uint16  // only report if delta > threshold (0 = report all)

STREAM_START — High Rate Analog

STREAM_START enables batched high-rate sampling. Multiple analog samples are packed into a single STREAM_DATA packet with no per-sample header. The host knows the schema from the subscribe request. This is the path for 200Hz IMU data over BLE without dropping.

// STREAM_START payload
pin_mask     : uint8   // bitmask of analog pins to stream
rate_hz      : uint16 // requested sample rate — device clamps to hardware max

// STREAM_DATA payload
// compact run of uint16 samples, ordered by pin index in pin_mask
// e.g. pin_mask = 0b00000111 (pins 0,1,2):
// [A0_s0][A1_s0][A2_s0][A0_s1][A1_s1][A2_s1] ...

Module System

Modules are optional C++ classes compiled into firmware. They register at boot with a name, version, and pin claim list. The host discovers them in HELLO_RESP and invokes them with MOD_CMD. Modules can push unsolicited MOD_EVENTs (encoder ticks, stepper completion, etc.).

Module Interface

class GraftModuleBase {
public:
  virtual const char*  name()    = 0;   // max 8 chars
  virtual uint8_t      version_major() = 0;
  virtual uint8_t      version_minor() = 0;
  virtual void         begin()   = 0;   // called once after device.begin()
  virtual void         handle(uint8_t cmd, GraftPayload& p, GraftContext& ctx) = 0;
  virtual uint8_t      pin_count() { return 0; }
  virtual uint8_t*     pins()      { return nullptr; }
};

Macro Pattern

GRAFT_MODULE(myModule) {

  GRAFT_ON_CMD(0x01) { // attach
    uint8_t pin    = payload.readUInt8();
    uint16_t minUs = payload.readUInt16();
    uint16_t maxUs = payload.readUInt16();
    servo.attach(pin, minUs, maxUs);
    ctx.ack();
  }

  GRAFT_ON_CMD(0x02) { // write angle
    uint8_t angle = payload.readUInt8();
    servo.write(angle);
    ctx.ack();
  }

  GRAFT_ON_DATASTREAM("setpoint") {
    float val = payload.readFloat32();
    pid.setTarget(val);
    ctx.ack();
  }
}

Transport Adapters

The transport layer is an abstract C++ interface: write(buf, len) and read(buf, maxlen). Device firmware picks one transport in setup(). The rest of the protocol is identical across all transports.

TransportClassNotes
UART/SerialGraftSerialCOBS-framed, 115200+ baud. Works on all Arduino-compatible devices.
USB CDCGraftUSBSerialSame framing, different underlying Serial class. Native USB on Leonardo, Pro Micro, etc.
BLEGraftBLEnRF52, ESP32. 20-byte MTU chunking. GATT notify (device→host) + write (host→device).
TCP/IPGraftTCPRaw socket. TCP framing eliminates need for COBS. ESP8266/ESP32/W5500.
MQTTGraftMQTTPublishes/subscribes to graft/{deviceId}/ topic tree. See §IoT Transport.
CLASPGraftCLASPBinary WebSocket relay. Channel = graft-{deviceId}. See §CLASP Transport.

Minimal Firmware Sketch

#include <Graft.h>
#include <GraftModuleServo.h>
#include <GraftModuleNeoPixel.h>

GraftSerial  transport(Serial, 115200);
GraftDevice  device("MyBoard", "1.0.0", transport);

void setup() {
  device.addModule(new GraftModuleServo());
  device.addModule(new GraftModuleNeoPixel());
  device.addDatastream("temperature", GRAFT_FLOAT32, "celsius", false);
  device.begin();
}

void loop() {
  device.poll(); // non-blocking, one packet per call
}

The library handles HELLO, ACK, subscriptions, CRC, COBS framing, and dispatches to modules. Custom modules are one class with a handle(cmd, data, len) method.

OTA & Provisioning

OTA Firmware Updates optional

Device announces OTA capability as a bit flag in HELLO_RESP. The host SDK handles chunking and retry. Implementation is platform-specific (Update.h on ESP32, InternalStorage on Nano 33 IoT) but the protocol framing is standardized.

// OTA sequence
host   → device : OTA_BEGIN   { total_bytes: uint32, sha256: 32 bytes }
device → host   : ACK

host   → device : OTA_CHUNK   { offset: uint32, data: N bytes } // repeat
device → host   : ACK         // or NAK 0x0F on chunk error

host   → device : OTA_FINALIZE
device → host   : ACK         // validates sha256, reboots
// device reboots — new HELLO_RESP will show updated firmware_version

BLE Provisioning Mode

If a device boots with no transport credentials (no WiFi SSID, no MQTT broker URL), it enters provisioning mode: a fixed BLE GATT service with a single write characteristic. The host SDK discovers and connects, pushes credentials as a structured payload, and the device reboots into normal GRAFT mode.

// BLE provisioning advertisement
Service UUID   : GRAFT-PROV-0001
Characteristic : GRAFT-PROV-WRITE

// Provisioning payload (host writes once)
wifi_ssid      : 32 bytes  // null-padded
wifi_pass      : 64 bytes  // null-padded
broker_url     : 64 bytes  // e.g. "mqtt://broker.local\0..."
auth_token     : 64 bytes  // broker credentials, null if not needed
device_id      : 32 bytes  // desired deviceId for topic routing

IoT Topology

Direct vs Brokered

// DIRECT — peer-to-peer [host SDK] ←——— serial / BLE ———→ [device firmware] // BROKERED — IoT topology [host SDK] ←——→ [MQTT broker / CLASP relay] ←——→ [device firmware] ↑ any other host dashboards Node-RED flows mobile apps // Device broadcasts retained HELLO_RESP on connect // All observers receive PIN_EVENTs with no coordination // Multiple hosts can subscribe simultaneously
21

MQTT Topic Schema

graft/
  {deviceId}/
    cmd/{typeHex}    ← host publishes here (e.g. cmd/11 = PIN_WRITE)
    evt/{typeHex}    ← device publishes here (e.g. evt/90 = PIN_EVENT)
    hello            ← HELLO_RESP binary, retain=true, QoS=1
    status           ← "online" | "offline", retain=true (LWT)
    meta             ← JSON capability summary for dashboards, retain=true
    ds/{name}/cmd    ← host writes to named datastream
    ds/{name}/evt    ← device pushes datastream value changes

LWT (Last Will and Testament)

Firmware registers LWT on broker connect. Device publishes "online" on clean connect and "offline" on clean disconnect. The broker publishes "offline" automatically if the device drops unexpectedly. Host SDKs subscribe to status and fire disconnect events automatically.

CLASP Transport

CLASP already has binary relay infrastructure. GRAFT over CLASP maps naturally without additional broker setup. Browser-native, no MQTT-over-WebSocket bridge required.

// CLASP transport mapping
CLASP channel  = "graft-{deviceId}"
CLASP msg type = GRAFT TYPE byte  // used as sub-channel selector
direction      = "req" (host→device) | "evt" (device→host)

// Firmware
GraftCLASP transport(
  "wss://relay.clasp.chat",
  "graft-workshop-uno-01",
  CLASP_TOKEN
);

// Host SDK
new CLASPTransport({
  relay:   'wss://relay.clasp.chat',
  channel: 'graft-workshop-uno-01',
  token:   process.env.CLASP_TOKEN,
})

Profiles & Fleet

Profiles live at the broker layer, not in firmware. A Profile is a JSON document that describes expected capabilities for a class of device. When a device connects and its HELLO_RESP fingerprint matches a Profile, the broker tags it with that profile. This enables fleet views, OTA rollouts, and bulk automation without any firmware changes.

{
  "profile":             "soil-sensor-v2",
  "match": {
    "firmware_name":    "SoilNode",
    "firmware_version": "2.*"
  },
  "expected_modules":    ["dht", "i2c_passthrough"],
  "expected_datastreams":["temperature", "humidity", "moisture"],
  "ota_channel":         "stable"
}

Device is still the source of truth. Profile is a broker-side grouping mechanism only. No firmware changes required to add a device to a fleet — it matches a Profile automatically.

Cross-Library Consistency Rules

These rules are normative — all host SDK implementations must follow them regardless of language idioms.

  • Transport is injectable. GraftDevice never constructs a transport internally. Always passed in. MockTransport is the same interface — test coverage is free.
  • Capabilities gate calls. Every method requiring a hardware feature checks HELLO_RESP first. Calling device.i2c(0) on a device with no I2C buses throws synchronously, not after a round-trip.
  • SEQ tracking is per-device, not per-transport. On transport reconnect, SEQ resets to 0 and a new HELLO is sent automatically. Outstanding promises from the previous connection are rejected.
  • Module methods wrap the module registry. No library hardcodes module command bytes in application-facing APIs. servo.write(90) emits MOD_CMD { module_id, cmd: 0x02, data: [90] }. Swapping module versions doesn't break the host SDK.
  • Streaming backpressure is idiomatic. Node.js streams, Python asyncio queues, Go channels with buffer, Rust Stream with poll_next. Device doesn't know or care. STREAM_DATA packets are QoS 0 — drops are acceptable.
  • Retained datastream sync on reconnect. After HELLO on reconnect, SDK requests retained state for all writable retained datastreams before application code runs.
  • GraftNAKError is typed. Every NAK produces a typed error with code (numeric), name (string), and seq (echoed sequence number). No opaque string errors.
